## Bringing It Together

Every module so far has asked you to apply one technique to the same recurring task you named back in Module 1. This capstone asks for all three artifacts at once, for that task — the same way a real team would actually use them: not as separate exercises, but as one connected system where each document makes the next one work.

As a reminder of what each artifact is for:

- The **instructions file** section holds the standing rule — the thing you'd otherwise repeat every session.
- The **spec** makes this one task deterministic — quantified, referencing a canonical example, with acceptance criteria that don't require judgment to check.
- The **workflow** is how you hand the spec to an agent without a wrong first guess turning into ten wrong files.

---

## The Line We Keep Returning To

> **An agent that builds fast on an ambiguous spec doesn't save you time — it just moves the correction to after the work is done, instead of before.**

Everything in this course is about paying that cost once, in writing, instead of repeatedly, in corrections.

<CodeLab title="Capstone: Your Recurring Task, Fully Specified">

Using the task you've carried through this course, produce all three documents. Keep each one short — a paragraph or a short list is enough; the goal is precision, not length.

**1. Instructions-file section**
Write the standing rule(s) an agent should follow every time this task comes up — a behavioral rule, a verification checklist, or a routing rule from Module 2. Include one line of reasoning for each rule, so someone can judge an edge case it doesn't explicitly cover.

**2. The spec**
Write the spec for one instance of this task. Find every vague quality word in your current mental version ("good," "comprehensive," "clean") and replace it with either a number/range or a named canonical reference — a real file, page, or example the agent should copy the structure of.

**3. The workflow**
Name the smallest first unit you'd apply this spec to, what you'd specifically check before batching the rest, and where the outcome gets recorded once it's confirmed (which changelog or knowledge-base file, and in what format).

If you get stuck on any of the three, that's useful information — it usually means the task itself needs to be scoped smaller before it's ready to hand to an agent at all.

</CodeLab>

<Quiz question="After finishing the capstone, a colleague asks: 'Why does this need three separate documents instead of one long, thorough prompt?' What's the best answer, based on this course?">
<div>

**Each document solves a different failure mode, and a single prompt can't hold all three roles at once: the instructions file prevents repeating standing rules every session, the spec removes ambiguity from one specific piece of work, and the workflow catches a wrong interpretation before it's repeated across many files — collapsing them into one prompt means losing whichever role gets crowded out.**

A single long prompt tends to mix standing conventions (which should persist across every future session) with task-specific detail (which is only relevant this one time) with process (how to hand off and verify the work) — and because nothing distinguishes the three, the standing conventions get re-typed every time, the task-specific detail gets buried in prose instead of being checkable, and there's no explicit point at which a first attempt gets verified before being applied broadly. Splitting them mirrors why this course covers them as three separate artifacts rather than "write a really good prompt": each has a different lifespan and a different job.

## Example: A Completed Capstone Deliverable

Here's what a finished version of the three artifacts looks like for one concrete recurring task — a single file combining all three, the same way you'd hand it to an agent or a teammate.

```markdown
# Task: Add a New Blog Post Template

## Standing rules (→ CLAUDE.md)
- Behavioral rule: if the post category isn't one of the 6 existing
  categories, ask before creating a new one — don't invent one silently
- Verification checklist: after adding a post, confirm it appears on the
  category index page and the RSS feed validates

## Spec
- Canonical reference: `posts/2026-06-01-example-post.md` — copy its
  frontmatter shape and section structure exactly
- Exact structure: frontmatter with `title`, `date`, `category` (one of
  the 6 existing categories), `summary` (1 sentence, under 160 chars);
  body has exactly 3 sections — Context, What Changed, Why It Matters
- Explicit rules: images go in `posts/images/<slug>/`, never inline as
  base64; no more than 1 image per section

## Acceptance criteria
- [ ] Frontmatter validates against the schema (`npm run validate:posts`)
- [ ] Post appears on its category index page after build
- [ ] `npm run build` produces a valid RSS entry for the new post
- [ ] `CHANGELOG.md` has a one-line `[feature]` entry naming the post

## Workflow
- Verify on: the first post only, reviewed against the checklist above
  before writing any further posts in the same batch
- Batch: additional posts only after the first is confirmed correct
- Log outcome in: `CHANGELOG.md` (root) — one line per post published;
  any new pattern (e.g. a new category getting approved) goes in
  `lessons-learned/content.md`
```
