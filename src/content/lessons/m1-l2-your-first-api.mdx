## Why APIs Matter

Every modern app is powered by APIs.

Your weather app, your social feed, your payment processor — they all speak to external services through a standardized request/response pattern. Learning to work with APIs unlocks an entire world of data and functionality you can build on top of.

---

## The Request / Response Cycle

At its core, an API call is simple:

1. You send a **request** (a URL + some parameters)
2. The server processes it and sends back a **response** (usually JSON)
3. You parse that JSON and use the data

```
Your App  →  HTTP Request  →  API Server
Your App  ←  JSON Response  ←  API Server
```

The tricky part isn't sending the request — it's handling what comes back gracefully.

---

## Fetching Data in JavaScript

The browser's built-in `fetch` API makes HTTP requests straightforward:

```js
const response = await fetch('https://api.example.com/data');
const data = await response.json();
console.log(data);
```

Two lines. That's the core of it.

But production code needs to handle errors, loading states, and unexpected response shapes — which is what this lesson covers.

<CodeLab title="Fetch and Display Real Data">

We'll use the free **JSONPlaceholder** API — no signup, no API key needed.

**Step 1** — Create a new file `src/api.js`:

```js
export async function fetchPosts() {
  const res = await fetch('https://jsonplaceholder.typicode.com/posts?_limit=5');
  if (!res.ok) throw new Error(`API error: ${res.status}`);
  return res.json();
}
```

**Step 2** — Call it in your component:

```js
import { fetchPosts } from './api.js';

async function loadPosts() {
  try {
    const posts = await fetchPosts();
    renderPosts(posts);
  } catch (err) {
    showError(err.message);
  }
}
```

**Step 3** — Render the results:

```js
function renderPosts(posts) {
  const container = document.getElementById('posts');
  container.innerHTML = posts
    .map(p => `<article><h3>${p.title}</h3><p>${p.body}</p></article>`)
    .join('');
}
```

Run it. You should see 5 post titles and bodies appear on screen.

</CodeLab>

<Pitfall title="Forgetting to Handle the Error State">

A `fetch()` call only throws on **network failure** — not on a bad HTTP status.

A `404` or `500` response will resolve the promise successfully. Your code will try to parse an error page as JSON and crash in a confusing way.

Always check `response.ok` before calling `.json()`:

```js
// ❌ Wrong — silently fails on 4xx/5xx
const data = await fetch(url).then(r => r.json());

// ✅ Correct — throws on bad status
const res = await fetch(url);
if (!res.ok) throw new Error(`HTTP ${res.status}`);
const data = await res.json();
```

</Pitfall>

<Quiz question="Why does fetch() not throw an error on a 404 response by default?">

<div>

**Because `fetch()` only rejects on network-level failures** — like DNS errors or the server being unreachable.

HTTP status codes (200, 404, 500) are valid, complete responses from the server's perspective. `fetch()` treats them all as successful network operations and resolves the promise.

It's your job to inspect `response.ok` (which is `true` for 200–299 status codes) and decide what to do with non-success statuses.

This is a frequent source of bugs for developers new to the Fetch API.

</div>
</Quiz>
