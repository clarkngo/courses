## Why Threat Modeling for AI Agents?

In traditional software, security is mostly about keeping unauthorized users out — firewalls, authentication, access control. In agentic AI, the threat surface is different. The agent itself can be manipulated into unsafe behavior. A user doesn't need to break in; they just need to say the wrong thing in the right way.

Threat modeling is how you think through that systematically, before something goes wrong. It answers four questions in order:

1. **What are we building?** — Map the system: components, data flows, external dependencies.
2. **What can go wrong?** — Identify threats using a framework like STRIDE.
3. **What are we going to do about it?** — Implement mitigations: validation, encryption, rate limits, logging.
4. **Did we do a good job?** — Test whether the controls actually work.

This lesson works through all four steps for the greeting agent you built in Lesson 1.

## The System: Data Flow Overview

Before identifying threats, you need a clear picture of what the system does and how data moves through it. The greeting agent has five key data flows:

1. **User → Agent (HTTP):** A user types a prompt in the browser. It travels to the `adk web` process on port 8000 — unencrypted, over localhost.
2. **Agent → `.env` file (file system read):** At startup, `agent.py` reads `GOOGLE_API_KEY` from the `.env` file.
3. **Agent → Gemini API (HTTPS):** `agent.py` sends the user prompt plus the API key credential to Google's servers.
4. **Gemini API → Agent (HTTPS):** The LLM response travels back to `agent.py` over HTTPS.
5. **Agent → Browser (HTTP):** The final response is sent back to the user's browser for display.

Each arrow in this picture is a potential attack surface. The question is: who controls that channel, and what happens if the data on it is tampered with?

## Trust Boundaries

A **trust boundary** is wherever data crosses from a lower-trust context into a higher-trust context — or vice versa. Every boundary is a point where you need to ask: *what happens if the data arriving here is malicious or wrong?*

For the greeting agent, at least three clear boundaries exist:

| Boundary | Components | What crosses it | Why it matters |
|---|---|---|---|
| **1** | Browser ↔ `adk web` | User prompt (text) | User input is untrusted — could contain injected instructions |
| **2** | `agent.py` ↔ Gemini API | Prompt + API key credential | External network; key must never appear in plaintext logs |
| **3** | `.env` file ↔ `agent.py` | `GOOGLE_API_KEY` value | File permissions determine who can read the credential |

A fourth boundary worth considering: is there a boundary between the `adk web` dev UI process and `agent.py`? What about between the Google Cloud project and the open internet? Every new component you add creates new boundaries to examine.

## Data Flow Diagrams

A **Data Flow Diagram (DFD)** makes trust boundaries visible. Two levels matter for threat modeling:

**Level 0 — Context Diagram**
The highest-level view. The entire system is one box. External entities (browser, Google Cloud) are rectangles. Arrows show what data flows in and out. Red dashed lines mark trust boundaries.

![Level 0 context diagram showing Web Browser User and Google Cloud as external entities flanking the greeting_agent system, with labeled data flows and red dashed trust boundaries](/courses/images/secure-agentic-ai/m1/level0-dfd.png)

**Level 1 — Decomposed View**
"Explodes" the system box into its internal components. This is where threats become specific.

![Level 1 decomposed DFD showing adk web and .env File inside the HOS01A Server boundary, agent.py inside the .venv boundary, with red external boundaries, a green internal boundary, and red X marks at threat intersections](/courses/images/secure-agentic-ai/m1/level1-dfd.png)

The boundaries and symbols:
- **Red dashed lines** (external) — left edge (browser side) and right edge (Google Cloud side). These are the outer walls.
- **Green dashed line** (internal) — runs between `.env File` and `agent.py`. This separates the data store holding the credential from the process that reads it.
- **Red X marks** — appear where a data flow arrow crosses a boundary. Two X marks appear in the Level 1 diagram: one where the `.env` read crosses the internal green boundary into `agent.py`, and one near the `adk web` → browser response flow crossing the external red boundary. Every X is a question that demands an answer: *what if the data here is malicious or intercepted?*

Note that `.env File` sits inside the **HOS01A Server** boundary alongside `adk web` — not inside the `.venv` boundary with `agent.py`. This placement matters: `.env` is readable by anything with access to the server filesystem, not just code running inside the virtual environment.

In security DFDs, **red dashed lines** mark external trust boundaries (like the moat around a castle). **Green dashed lines** mark internal boundaries (the locked rooms inside). Where a data flow arrow crosses a trust boundary line, you mark the intersection with a **red X** — these are **threat intersections**, the exact points where an attacker could intercept, inject, or corrupt data. They are the most important locations in the diagram for a security engineer: every red X is a question that demands an answer.

## STRIDE Analysis

**STRIDE** is a threat categorization framework. Each letter is a threat category with a corresponding security goal:

| Threat | Security Goal | Agentic AI Example | OWASP 2026 | Risk |
|---|---|---|---|:---:|
| **S**poofing | Authenticity | A malicious agent masquerades as a "Boss Agent" to issue unauthorized commands to worker agents | ASI09: Improper Input Handling | High |
| **T**ampering | Integrity | Indirect prompt injection: an attacker hides instructions in a document the agent reads, hijacking its goal | ASI01: Agent Goal Hijacking | High |
| **R**epudiation | Non-repudiability | An agent deletes a database record but the system fails to log which agent did it or why | ASI10: Insufficient Monitoring | Medium |
| **I**nformation Disclosure | Confidentiality | An agent leaks its system prompt, API keys, or private user data during a conversation | ASI07: Sensitive Information Disclosure | High |
| **D**enial of Service | Availability | An attacker triggers a recursive loop (Agent A calls Agent B calls Agent A) until API credits are exhausted | ASI08: Unbounded Consumption | Medium |
| **E**levation of Privilege | Authorization | A user tricks a "read-only" agent into using a write tool, or overrides its system instruction via a crafted prompt | ASI06: Excessive Agency | High |

The OWASP codes come from the **OWASP Top 10 for Agentic Applications (2026)**. You'll work through these in detail in Lesson 3 — for now, notice that every High-risk STRIDE category maps directly to a named OWASP item. These aren't coincidental: OWASP derived its list specifically from the kinds of agent-level threats STRIDE surfaces.

## Mitigations

For each threat, a mitigation either reduces its **likelihood** (prevention) or reduces its **impact** (detection or response). Here are three worked examples for the greeting agent:

### Prompt Injection (Elevation of Privilege)

**Threat:** An attacker crafts a user prompt that overrides the agent's system instruction, causing the agent to behave outside its defined role.

**Mitigation:** Add a content safety check before passing input to the LLM. Use Gemini's built-in safety settings to block harmful content categories.

**Type:** Prevention

---

### API Key Disclosure (Information Disclosure)

**Threat:** The `GOOGLE_API_KEY` in the `.env` file is accidentally committed to a public GitHub repository.

**Mitigation:** Confirm `.env` is listed in `.gitignore`. Enable GitHub secret scanning on your repo. Rotate the key immediately if exposure is detected — a leaked key cannot be "un-leaked," only replaced.

**Type:** Prevention + Detection

---

### Quota Exhaustion (Denial of Service)

**Threat:** An attacker or a runaway loop makes excessive Gemini API calls, exhausting the project's quota and making the agent unavailable.

**Mitigation:** Set a budget alert and API quota limit in the Google Cloud Console. Add a rate limit on the `adk web` endpoint if it is ever exposed beyond localhost.

**Type:** Prevention

<Pitfall title="The Dev UI Is Not Production-Safe">

`adk web` binds to `localhost:8000` by default. That's fine during development. But if you expose that port — through a Codespace port-forwarding rule, a cloud VM, or a misconfigured firewall — any network user can send prompts to your agent with no authentication.

The dev UI was built for local debugging. Before exposing any agent endpoint publicly, you need authentication, HTTPS, and rate limiting. None of those come with `adk web` out of the box.

</Pitfall>

<Quiz question="An agent reads a webpage and follows an instruction hidden in the page content, even though that instruction contradicts its system prompt. Which STRIDE category does this fall under?">

<div>

**Elevation of Privilege** — and also **Tampering**.

This is **indirect prompt injection**: an attacker embeds malicious instructions in external content (a webpage, a document, an email) that the agent is expected to read. When the agent processes the content, it interprets the hidden instructions as legitimate commands and changes its behavior.

The E in STRIDE covers authorization violations — the agent was authorized to read the page, but not to let the page rewrite its goals. The T covers the integrity violation: the agent's planned behavior was tampered with by data it trusted too much.

In later modules, as your agents gain the ability to browse the web and read files, this threat becomes one of the most important to design against.

</div>
</Quiz>
