## What Is an AI Agent?

A basic LLM just responds to a single prompt. An **AI agent** (or LLM agent) goes further — it uses a large language model as its core "brain" to pursue complex goals across multiple steps.

Here's what separates an agent from a plain chatbot:

**1. Reasoning and Planning**
The agent doesn't just react — it can break a problem into sub-tasks and formulate a plan. Internally, this often uses "chain-of-thought" reasoning: think step-by-step before acting.

**2. Memory**
- *Short-term memory* — the ongoing conversation context. The agent remembers what was said earlier in the same session.
- *Long-term memory* — information stored across sessions, often in a vector database, so the agent can recall past interactions and preferences.

**3. Tools**
Agents can call external tools to go beyond language: run web searches, query databases, execute code, call APIs. The tool call's result feeds back into the agent's next decision.

**4. Autonomous Loop**
Rather than one prompt → one response, agents run a repeating cycle:

1. **Perceive** — receive input or observe a change
2. **Think** — reason through the goal, form a plan
3. **Act** — call a tool, generate a response, or delegate to another agent
4. **Observe** — check the result
5. **Reflect** — if the goal isn't met, adjust and loop again

This sense-think-act loop is what makes agents capable of handling tasks that require multiple steps or real-world interaction.

## Google Agent Development Kit (ADK)

**ADK** is Google's open-source, code-first framework for building AI agents in Python. It gives you:

- **LlmAgent** — the standard agent type; uses an LLM to reason, plan, and call tools dynamically
- **Workflow agents** — `SequentialAgent`, `ParallelAgent`, `LoopAgent` for deterministic flows that don't need LLM decision-making
- **A local dev UI** — launch `adk web` and get a browser interface to chat with your agent, inspect events, and view trace logs
- **Deployment path** — agents built with ADK can be containerized and deployed to Google Cloud Run or Vertex AI Agent Engine

ADK is optimized for Gemini models but supports other LLMs through LiteLLM.

## Step 1: Set Up a Virtual Environment

A virtual environment keeps your project's Python packages isolated from your system. Always use one.

```bash
# Add .venv to .gitignore first — it can be hundreds of megabytes
echo .venv >> .gitignore

# Create the virtual environment
python -m venv .venv

# Activate it (macOS/Linux / GitHub Codespaces)
source .venv/bin/activate

# Windows CMD
# .venv\Scripts\activate.bat

# Windows PowerShell
# .venv\Scripts\Activate.ps1
```

Once activated, your terminal prompt will show `(.venv)` to confirm you're inside the environment.

Now install Google ADK:

```bash
echo google-adk[database]==1.6.1 >> requirements.txt
pip install -r requirements.txt
```

The `[database]` extra pulls in drivers for ADK's database integrations (used in later modules via the Model Context Protocol toolbox). Pinning `==1.6.1` ensures everyone on the project gets the same behavior.

---

## Step 2: Create the Agent Project

```bash
mkdir greeting_agent
touch greeting_agent/__init__.py
touch greeting_agent/agent.py
touch greeting_agent/.env
```

**`__init__.py`** — makes the folder a Python package and imports the agent:

```python
from . import agent
```

**`agent.py`** — the core of your agent:

```python
from google.adk.agents import Agent

root_agent = Agent(
    name="greeting_agent",
    model="gemini-2.0-flash",
    description="Greeting agent",
    instruction="""
        You are a helpful assistant that greets the user.
        Ask for the user's name and greet them by name.
    """,
)
```

<CodeLab title="What Each Parameter Does">

| Parameter | Required | Purpose |
|---|---|---|
| `name` | Yes | Unique identifier. In multi-agent systems, other agents use this name to delegate tasks. |
| `model` | Yes | Which LLM powers the agent. `gemini-2.0-flash` is fast and cost-efficient. |
| `description` | No | A short summary of what this agent does. **Critical in multi-agent systems** — other agents read this to decide whether to route a task here. |
| `instruction` | Yes | The system prompt. This is the most powerful parameter — it defines the agent's role, personality, constraints, and tool-use guidelines. |

The `instruction` field is where your security posture lives. In later modules, this is where you'll add constraints like "never reveal internal data" or "always validate user input before acting."

</CodeLab>

## Step 3: Configure Your API Key

Open `greeting_agent/.env` and add:

```
GOOGLE_GENAI_USE_VERTEXAI=FALSE
GOOGLE_API_KEY=your_actual_api_key_here
```

`GOOGLE_GENAI_USE_VERTEXAI=FALSE` tells the ADK to use the public Gemini API directly (via Google AI Studio) rather than Vertex AI. This is the simpler path for development.

To get an API key:
1. Go to [Google AI Studio](https://aistudio.google.com/apikey)
2. Sign in with your Google account
3. Click **Get API key** → **Create API key**
4. Select or create a Google Cloud project
5. Copy the key and paste it into your `.env` file

<Pitfall title="Never Commit Your API Key">

The `.env` file holds a real credential. If it ends up in a public GitHub repo, anyone can use your key — and you pay the bill.

Two rules:
1. Add `.env` to `.gitignore` (in addition to `.venv`)
2. Never paste an API key directly into `agent.py` or any file that gets committed

This isn't just good practice — it's the foundation of **secrets management**, a core topic in secure agentic AI.

</Pitfall>

## Step 4: Run Your Agent

Navigate to the **parent directory** of your agent folder (not inside `greeting_agent/`), then launch the dev UI:

```bash
adk web
```

Open the URL shown in your terminal — usually `http://localhost:8000`.

In the top-left dropdown, select **greeting_agent**. Then try these prompts:

- `"How do you say hello in Spanish?"`
- `"What's a formal greeting in Japanese?"`
- `"Tell me how to greet someone in French."`

**Exploring the Events panel:**
The left sidebar has an **Events** tab. Click any event to inspect the raw function calls and model responses — this is how you see exactly what the LLM decided to do at each step.

Click **Trace** on any event to see latency for each function call. You'll use this in later modules to spot slow or expensive operations.

<Quiz question="What is the purpose of the `instruction` parameter in an ADK Agent?">

<div>

The `instruction` is the system prompt — it defines the agent's role, personality, and behavioral constraints. It's the single most important parameter for controlling what an agent will and won't do.

In a security context, the `instruction` is also where you encode guardrails: what topics the agent should refuse, how it should handle unexpected input, and when it should escalate to a human. A weak or vague instruction is one of the most common sources of unsafe agent behavior.
