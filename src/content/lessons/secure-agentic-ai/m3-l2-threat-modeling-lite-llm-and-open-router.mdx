## What Changed from Module 2?

In Module 2, `agent.py` called the Gemini API directly — one hop to an LLM provider you chose. The `dad_joke_agent` adds two new components in that path:

1. **LiteLLM** — an in-process routing layer that translates API calls before they leave the server
2. **OpenRouter** — a third-party external gateway that forwards requests to the actual LLM provider

This single architectural change creates two new trust boundaries that didn't exist before. The threat model grows accordingly.

---

## The System: Data Flow Overview

The `dad_joke_agent` has seven data flows. Flows 3–6 are restructured compared to Module 2 — the LLM is no longer a direct dependency:

1. **User → Agent (HTTP):** The user's prompt travels from the browser to `adk web` on port 8000.
2. **Agent → `.env` file (file system read):** At startup, `agent.py` reads `OPENROUTER_API_KEY` from the `.env` file.
3. **Agent → LiteLLM (local):** `agent.py` sends the user prompt and tool definitions to the LiteLLM routing layer, running in the same process.
4. **LiteLLM → OpenRouter (HTTPS):** LiteLLM translates the request and forwards it to the OpenRouter API over HTTPS.
5. **OpenRouter → LiteLLM → Agent (HTTPS):** OpenRouter routes to GPT-4.1, receives the tool call request, and returns it via LiteLLM to `agent.py`.
6. **Agent → Tool → Agent → LiteLLM → OpenRouter (local + HTTPS):** `agent.py` executes `get_dad_joke()` locally, sends the result back through LiteLLM to OpenRouter for the final response.
7. **Agent → Browser (HTTP):** The final response is sent back to the user.

The key difference from Module 2: **your prompts now pass through a third party you don't control.** Every word the user types, and every tool result, travels through OpenRouter's infrastructure before the LLM sees it.

## Trust Boundaries

Five trust boundaries exist in this system. Boundaries 4 and 5 are new:

| Boundary | Components | What crosses it | Why it matters |
|---|---|---|---|
| **1** | Browser ↔ `adk web` | User prompt (text) | User input is untrusted — could contain injected instructions |
| **2** | `agent.py` ↔ Gemini API (via OpenRouter) | Prompt + tool definitions + API key | External network; credentials must not be exposed in transit |
| **3** | `.env` file ↔ `agent.py` | `OPENROUTER_API_KEY` | File permissions determine who can read the credential |
| **4 (NEW)** | LLM reasoning output ↔ Tool Execution Layer | Tool call arguments and tool return values | GPT-4.1 generates tool arguments from user input — prompt injection can propagate into local function calls |
| **5 (NEW)** | `agent.py` ↔ LiteLLM Layer | Prompt, tool definitions, tool results | LiteLLM is an in-process translation layer, but its output crosses to an external third-party gateway (OpenRouter). This routing boundary didn't exist when calling Gemini directly. |

<CodeLab title="The LiteLLM Routing Layer as a New Trust Boundary">

Boundary 5 is subtle but important. LiteLLM runs *inside* your process — it's not a network call. But it exists at the edge between your code and an external service you don't control.

When `agent.py` calls `LiteLlm(...)`, the library:
1. Reformats your prompt to match OpenRouter's expected schema
2. Attaches your API key
3. Makes an HTTPS request to `openrouter.ai`

Step 3 is where the trust boundary is crossed. Every prompt your users type, and every tool result your agent generates, leaves your infrastructure and enters OpenRouter's. From there, OpenRouter routes to GPT-4.1 or whichever provider you've configured.

**What this means in practice:**
- OpenRouter's data retention policy governs whether your prompts are logged or used for training
- An outage at OpenRouter makes your agent unavailable, regardless of how reliable your own infrastructure is
- A breach at OpenRouter could expose your prompt history, API key, or tool results
- You cannot audit what happens to your data inside OpenRouter's pipeline

This is the fundamental trade-off of third-party gateway services: you gain model flexibility and a unified API, but you lose visibility and control over data in transit.

</CodeLab>

## Mitigations

### Tool Parameter Injection (Tampering / Elevation of Privilege)

**Threat:** An attacker crafts a prompt that causes GPT-4.1 via OpenRouter to invoke `get_dad_joke()` with an unintended argument.

**Mitigation:** Validate tool arguments inside the function before execution. Log all invocations with arguments and timestamps.

**Type:** Prevention + Detection

---

### Third-Party Gateway Risk (Information Disclosure / Denial of Service)

**Threat:** All LLM requests are routed through OpenRouter, a third-party service. A breach or policy change at OpenRouter could expose prompt data or disrupt service.

**Mitigation:** Review OpenRouter's privacy and data retention policy before sending sensitive data. Avoid including PII in prompts. Implement fallback logic in LiteLLM to switch providers if OpenRouter is unavailable.

**Type:** Prevention + Detection

---

### API Key Information Disclosure

**Threat:** The `OPENROUTER_API_KEY` in the `.env` file is accidentally committed to a public GitHub repository.

**Mitigation:** Confirm `.env` is in `.gitignore`. Enable GitHub secret scanning. Rotate the key immediately if exposure is detected.

**Type:** Prevention + Detection

---

### Tool Call Repudiation

**Threat:** Tool invocations — which tool, which arguments, which timestamp — are not logged, making the agent's actions unauditable.

**Mitigation:** Add structured logging inside each tool function. Store logs in a tamper-evident location separate from the agent process.

**Type:** Detection

---

### Denial of Service via Tool Loop

**Threat:** An adversarial prompt causes the agent to invoke `get_dad_joke()` repeatedly, exhausting OpenRouter credits.

**Mitigation:** Set a budget alert and rate limit in the OpenRouter dashboard. Implement a maximum tool-call count per session.

**Type:** Prevention

<Pitfall title="Your Prompts Are Now Third-Party Data">

With direct Gemini API calls, your prompt data goes to Google under Google's terms. With OpenRouter, it goes to OpenRouter first, then to whichever downstream provider OpenRouter routes to.

That means you're agreeing to two data processing relationships, not one — and both matter for compliance if you're handling user data that has regulatory implications (GDPR, HIPAA, FERPA). Before using any AI gateway in production, read its data processing agreement and understand what it logs, how long it retains it, and whether it can be used to improve models.

For a dad joke agent, this is a low-stakes concern. For an agent that handles student records or medical queries, it's a blocking issue.

</Pitfall>

<Quiz question="OpenRouter suffers a 2-hour outage. Which STRIDE category does this represent, and what mitigation pattern addresses it?">

<div>

**Denial of Service** — the availability of your agent depends entirely on a third-party service you don't control.

The mitigation pattern is **provider fallback**: configure LiteLLM to switch to a backup provider if the primary is unavailable. LiteLLM supports this natively through its routing and fallback configuration. For example, you can set OpenRouter as primary and a direct Gemini API call as fallback — if OpenRouter returns a 5xx error or times out, LiteLLM automatically retries with the backup provider.

```python
model = LiteLlm(
    model="openrouter/openai/gpt-4.1",
    api_key=os.getenv("OPENROUTER_API_KEY"),
    fallbacks=["gemini/gemini-2.0-flash"],
)
```

This is the same defense-in-depth principle as having a secondary ISP: single-provider dependency is a single point of failure. The more your agent's availability matters, the more important provider redundancy becomes.

</div>
