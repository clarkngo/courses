## Why a Standard?

The controls built in Modules 2–5 address specific threats in a specific architecture. A standard provides three things those controls alone cannot:

1. **Completeness assurance** — have we missed a category of control that a thoughtful adversary would exploit?
2. **Auditability** — can an independent party verify that our controls are adequate without reading every line of code?
3. **Regulatory alignment** — in regulated industries (nuclear, power grid, water treatment), a standard provides the legal basis for demonstrating due diligence.

ISA/IEC 62443 is the only internationally recognized standard specifically for industrial automation and control system cybersecurity. Understanding its structure lets you map your work to audit requirements efficiently.

---

## The Document Family

ISA/IEC 62443 is a collection of standards, not a single document. Knowing which documents apply to your role:

### Series 1: General (Concepts and Vocabulary)

| Document | Title | Use When |
|---|---|---|
| 62443-1-1 | Terminology, concepts, and models | Starting a site assessment — establishes shared vocabulary |
| 62443-1-2 | Master glossary | Resolving definitional disputes in assessments |
| 62443-1-3 | System security compliance metrics | Measuring current vs. target security level |
| 62443-1-4 | IACS security lifecycle | Planning a multi-year security program |

### Series 2: Policies and Procedures (Asset Owner / Operator)

| Document | Title | Use When |
|---|---|---|
| 62443-2-1 | Requirements for an IACS security management system | Building a security program (governance layer) |
| 62443-2-2 | Implementation guidance for an IACS CSMS | Implementing 62443-2-1 requirements operationally |
| 62443-2-3 | Patch management in the IACS environment | Designing the OT patch process |
| 62443-2-4 | Requirements for IACS solution suppliers | Vetting vendors and system integrators |

### Series 3: System Requirements (System Integrator)

| Document | Title | Use When |
|---|---|---|
| 62443-3-2 | Security risk assessment and system design | Performing the zone and conduit risk assessment (Module 1) |
| 62443-3-3 | System security requirements and security levels | The core technical requirements — 51 FRs mapped to SLs |

### Series 4: Component Requirements (Product Supplier)

| Document | Title | Use When |
|---|---|---|
| 62443-4-1 | Secure product development lifecycle | Evaluating whether a PLC vendor follows secure SDLC |
| 62443-4-2 | Technical security requirements for IACS components | Evaluating specific device capabilities (e.g., does this PLC support authentication?) |

For most security engineers working at an asset owner or system integrator level, the most relevant documents are **62443-3-2** (risk assessment), **62443-3-3** (technical requirements), and **62443-2-1** (security management system).

## The Security Lifecycle

62443-1-4 defines a lifecycle model for industrial security — not a one-time assessment, but an ongoing process:

```
Assess → Design → Implement → Maintain → Monitor → Respond → Review → Assess...
```

This lifecycle maps to the course modules:

| Lifecycle Phase | Course Coverage |
|---|---|
| **Assess** | Module 1: Risk register, zone/conduit identification, SL assignment |
| **Design** | Module 2: WireGuard network design; Module 3: IT security architecture |
| **Implement** | Modules 2–5: WireGuard, Vault, Zeek, Suricata, Modbus proxy |
| **Maintain** | Module 3: Vault key rotation, pipeline signing; Module 4: Zeek baseline updates |
| **Monitor** | Module 4: Zeek behavioral logging; Suricata alert monitoring |
| **Respond** | Module 5: Circuit breaker, alert integration, operator reset procedure |
| **Review** | Module 6: Compliance mapping, audit evidence, SL-A vs SL-T assessment |

## The 7 Foundational Requirements Revisited

In Module 1, you saw the 7 FRs at a high level. Now that the controls are implemented, map each one to specific artifacts:

| FR | Category | Control Implemented | Where |
|---|---|---|---|
| FR 1 | Identification & Authentication | Vault AppRole, MFA on jump server, WireGuard key-based auth | M2, M3 |
| FR 2 | Use Control | nftables ACLs, OAuth scopes, Vault policies, vendor time-limited access | M2, M3 |
| FR 3 | System Integrity | Signed build pipeline, firmware signing with cosign | M3 |
| FR 4 | Data Confidentiality | WireGuard encryption, TLS 1.3 on historian API | M2, M3 |
| FR 5 | Restricted Data Flow | Zone/conduit design, OT DMZ with dual firewalls, nftables per-protocol rules | M1, M2 |
| FR 6 | Timely Response to Events | Zeek notice.log → SIEM, Suricata alerts, circuit breaker paging | M4, M5 |
| FR 7 | Resource Availability | Fail-open Suricata, proxy watchdog, reads-allowed during circuit open | M4, M5 |

This mapping is the core of your compliance documentation. An auditor working through 62443-3-3 SR by SR should be able to find the evidence for each requirement in your documented controls.

## ISA/IEC 62443 vs. Other Standards

How does 62443 relate to other standards you may encounter?

| Standard | Scope | Relationship to 62443 |
|---|---|---|
| **NIST CSF** | General enterprise security | Complementary — CSF functions (Identify, Protect, Detect, Respond, Recover) map well to 62443 lifecycle |
| **IEC 61511** | Safety instrumented systems (functional safety) | Different standard — 62443 covers cyber, 61511 covers functional. Must address both independently |
| **NERC CIP** | North American bulk electric system | Mandatory for grid operators; 62443 compliance helps meet NERC CIP requirements but doesn't fully replace them |
| **NIS2 Directive** | EU critical infrastructure (including OT) | 62443 compliance is widely accepted as evidence of NIS2 technical measures |
| **IEC 62351** | Power systems security protocols | Specific to power sector protocols (DNP3, IEC 61850); complements 62443 FR 1, FR 4 for utilities |

<Pitfall title="62443 Compliance Is a Point-in-Time Statement">

An ISA/IEC 62443 compliance assessment says "as of this date, these controls meet these requirements." It does not guarantee ongoing compliance. A firewall rule deleted six months after the assessment, a key rotation skipped, or a new OT device added without zone assignment can silently create a gap.

The compliance program requires a continuous monitoring component (FR 6) and a regular review cadence (62443-2-1 Section 4.3.2) to remain valid. Many organizations treat compliance as a one-time audit and are surprised when an incident exposes a gap that the controls were supposed to close.

</Pitfall>

<Quiz question="An OT engineer asks why the team needs to document compliance with ISA/IEC 62443 SR 2.3 (use control for portable and mobile devices) when no laptops are allowed on the OT network. Is SR 2.3 relevant?">

<div>

**Yes — SR 2.3 is relevant, and the assertion that no laptops are allowed needs to be verified and documented.**

SR 2.3 requires that a policy exists for portable and mobile devices in the OT environment, and that the policy is enforced. The fact that laptops are prohibited is the policy — but compliance requires:

1. **Documentation**: the policy must be written down, approved, and accessible
2. **Technical enforcement**: "no laptops allowed" should be enforced technically, not just procedurally. Options include: locking unused switch ports, enabling 802.1X port authentication on OT switches, or using USB port control software on OT workstations
3. **Evidence**: an auditor needs to see evidence that the technical enforcement works — not just a policy document

Additionally, "no laptops" often turns out to be inaccurate when physically inspected. Vendor technicians frequently connect laptops for maintenance. The policy needs to address what happens during vendor access (temporary exception with logging, as per the vendor access process in Module 2) rather than implying it never happens.

A gap in SR 2.3 is more likely to be discovered during a physical inspection than a network scan. Walk the plant floor before declaring compliance.

</div>
</Quiz>
