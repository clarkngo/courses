## From Design to Measurement

Security Level Target (SL-T) is a design intent. Security Level Achieved (SL-A) is a measurement of what's actually implemented. The gap between them is your risk exposure — and closing that gap is the purpose of your remediation roadmap.

This lesson performs the SL-T vs. SL-A comparison for the architecture built throughout this course.

## Compensating Control Documentation

When a native control is not technically feasible (e.g., enabling authentication on a 20-year-old Modbus-only PLC), ISA/IEC 62443 allows **compensating controls** — alternative controls that reduce the risk to an acceptable level. Document them precisely:

```yaml
# compensating-controls.yaml
compensating_controls:
  - id: CC-001
    sr_reference: "62443-3-3 SR 1.1 — Identification and Authentication"
    zone: "Field Control (OT Network)"
    reason_native_not_feasible: >
      Legacy PLCs (Siemens S7-300 series) use Modbus TCP which has no native
      authentication. Firmware upgrade to add authentication not available from
      vendor. Hardware replacement budgeted for 2028.
    compensating_control: >
      Modbus proxy (modbus_proxy.py) validates source IP against authorized
      writer whitelist before forwarding commands to PLC. Network segmentation
      (VLAN isolation + nftables) ensures only the proxy host can reach PLC port 502.
      WireGuard tunnel (Module 2) authenticates all traffic entering the OT VLAN.
    risk_reduction: >
      An attacker who compromises an IT host cannot send Modbus commands directly
      (VLAN isolation). An attacker on the OT VLAN who is not the proxy host is
      blocked by nftables. The compensating control achieves equivalent risk
      reduction to native authentication for the defined threat actors (SL 2).
    approval:
      approver: "Plant Manager — J. Smith"
      date: "2026-06-01"
      review_date: "2027-06-01"
    status: approved
```

This document is signed by the plant manager and becomes part of the audit package.

<CodeLab title="Automated SL Gap Report">

This script compares your compliance traceability YAML against expected SL requirements and generates a gap report:

```python
#!/usr/bin/env python3
# sl-gap-report.py

import yaml
import sys

with open('compliance-traceability.yaml') as f:
    traceability = yaml.safe_load(f)

target_sl = traceability['target_security_level']
gaps = []

for fr_name, srs in traceability['foundational_requirements'].items():
    for sr_name, sr_data in srs.items():
        required_sl = sr_data.get('sl_required', 1)
        achieved = sr_data.get('status', 'unknown')
        if achieved != 'compliant' or required_sl > target_sl:
            gaps.append({
                'sr': sr_name,
                'required_sl': required_sl,
                'status': achieved,
                'owner': sr_data.get('owner', 'unassigned'),
            })

if gaps:
    print(f"SL GAP REPORT — Target SL: {target_sl}")
    print(f"Gaps found: {len(gaps)}\n")
    for gap in gaps:
        print(f"  {gap['sr']}")
        print(f"    Required SL: {gap['required_sl']} | Status: {gap['status']}")
        print(f"    Owner: {gap['owner']}\n")
    sys.exit(1)
else:
    print(f"All SRs compliant at SL {target_sl}.")
    sys.exit(0)
```

Integrate this script into your CI/CD pipeline — if a compliance gap is introduced (e.g., a control is removed without a replacement), the pipeline fails and requires a review before merge.

</CodeLab>

<Pitfall title="Don't Mistake Zone SL for Component SL">

A zone's SL-A is bounded by its weakest component's SL. A perfectly configured WireGuard gateway in an OT DMZ zone doesn't bring the zone to SL 3 if the SCADA server in the same zone runs Windows XP with no antivirus and no patch management.

62443-4-2 defines component-level requirements (SL-C) for individual devices. A zone can only achieve SL N if all its components can demonstrate SL-C N. Audit each component independently before claiming zone-level compliance.

This is why legacy PLCs and unpatched SCADA servers frequently cap a zone at SL 1 regardless of how sophisticated the perimeter controls are.

</Pitfall>

<Quiz question="The Field Control zone has SL-T = 2 but SL-A = 1 due to legacy PLCs without native authentication. The plant manager asks whether upgrading the surrounding controls (better proxy, stronger Zeek scripts) can bring SL-A to 2. What is the correct answer?">

<div>

**Partially yes — compensating controls can raise the effective protection level, but the zone's formal SL-A cannot exceed what the weakest component can demonstrate.**

The correct path:

1. **Document compensating controls** per the template shown in this lesson. The proxy's source IP validation, combined with network isolation, provides equivalent protection to native authentication for the defined threat actor (intentional, simple means, SL 2).

2. **Submit compensating control documentation for approval** — the plant manager must sign off. This documents that the residual risk is accepted and the compensating controls are formally recognized.

3. **Record the gap** in the compliance traceability matrix with status `compensating_control_approved` rather than `compliant`. The zone's SL-A is annotated as "SL 1 native, SL 2 with approved compensating controls."

4. **Plan for native remediation** — the compensating control documentation should include a timeline for hardware replacement or firmware upgrade that provides native authentication.

An auditor will accept this approach if the documentation is complete and the approver is appropriate (plant manager for safety-impacting gaps). They will not accept undocumented reliance on compensating controls, or compensating controls approved only by the security team.
