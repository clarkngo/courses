## What RAG Solves

An LLM's training data has a cutoff date. It doesn't know your organization's runbooks, your custom threat models, or your equipment-specific manuals. If Agent B relied solely on the LLM's built-in knowledge, its recommendations would be generic and potentially wrong for your environment.

**Retrieval-Augmented Generation (RAG)** solves this by:

1. Storing your internal documentation in a vector database
2. At inference time, finding the most relevant documents for the current incident
3. Including those documents in the LLM prompt as grounding context

The LLM's job changes from "know everything" to "reason over what you're given."

---

## RAG Architecture for Cyber AI

```
Internal Docs (PDFs, Markdown, text)
          │
          ▼
    [Document Loader]      ← knowledge_base/loader.py
          │
    chunk + clean
          │
          ▼
    [Embedding Model]      ← sentence-transformers (local)
          │
    vector representation
          │
          ▼
    [ChromaDB Collection]  ← persistent on disk
          │
          │  ← at inference time ─────────────────────────────┐
          │                                                    │
    [Semantic Search]  ←─ incident context (from Agent A)     │
          │                                                    │
    top-k documents                                           │
          │                                                    │
          ▼                                                    │
    [Agent B prompt]  ←────────────────────────────────────────┘
```

## Part 1: Document Loader

```python
# knowledge_base/loader.py
import os
import re
import chromadb
from chromadb.utils import embedding_functions

CHROMA_PATH = "./chroma_db"
COLLECTION_NAME = "secops_kb"

def _chunk_text(text: str, chunk_size: int = 500, overlap: int = 50) -> list[str]:
    """Split text into overlapping chunks."""
    words = text.split()
    chunks = []
    i = 0
    while i < len(words):
        chunk = " ".join(words[i : i + chunk_size])
        chunks.append(chunk)
        i += chunk_size - overlap
    return chunks

def load_documents(docs_dir: str = "./docs") -> None:
    """Load all .txt and .md files from docs_dir into ChromaDB."""
    client = chromadb.PersistentClient(path=CHROMA_PATH)
    ef = embedding_functions.SentenceTransformerEmbeddingFunction(
        model_name="all-MiniLM-L6-v2"
    )
    collection = client.get_or_create_collection(
        name=COLLECTION_NAME,
        embedding_function=ef,
    )

    doc_id = 0
    for filename in os.listdir(docs_dir):
        if not filename.endswith((".txt", ".md")):
            continue
        filepath = os.path.join(docs_dir, filename)
        with open(filepath) as f:
            raw = f.read()

        chunks = _chunk_text(raw)
        for i, chunk in enumerate(chunks):
            collection.upsert(
                ids=[f"{filename}-chunk-{i}"],
                documents=[chunk],
                metadatas=[{"source": filename, "chunk": i}],
            )
            doc_id += 1

    print(f"Loaded {doc_id} chunks from {docs_dir} into ChromaDB.")
```

<CodeLab title="Chunk Size and Overlap">

Chunking splits long documents into pieces small enough to fit in a prompt. The tradeoffs:

- **Smaller chunks** → more precise retrieval, but context can be cut off mid-thought
- **Larger chunks** → more context per result, but you retrieve fewer unique passages
- **Overlap** → repeated words at chunk boundaries ensure a sentence that straddles a boundary appears in at least one chunk

`500 words / 50 word overlap` is a reasonable starting point for technical documentation. Tune based on your retrieval quality.

</CodeLab>

## Part 3: Semantic Query Function

```python
# knowledge_base/retriever.py
import chromadb
from chromadb.utils import embedding_functions

CHROMA_PATH = "./chroma_db"
COLLECTION_NAME = "secops_kb"

_client = None
_collection = None

def _get_collection():
    global _client, _collection
    if _collection is None:
        _client = chromadb.PersistentClient(path=CHROMA_PATH)
        ef = embedding_functions.SentenceTransformerEmbeddingFunction(
            model_name="all-MiniLM-L6-v2"
        )
        _collection = _client.get_collection(
            name=COLLECTION_NAME, embedding_function=ef
        )
    return _collection

def retrieve(query: str, n_results: int = 3) -> list[str]:
    """Return top-n relevant document chunks for a query."""
    collection = _get_collection()
    results = collection.query(query_texts=[query], n_results=n_results)
    return results["documents"][0]  # List of chunk strings
```

<Pitfall title="Stale Knowledge Base">

A RAG system is only as good as its documents. If the runbooks in ChromaDB are a year old, Agent B's recommendations will reflect old procedures.

Build a re-indexing trigger into your pipeline: run `load_documents()` whenever files in `docs/` change (e.g., via a file watcher or a CI step). Use `upsert` (not `add`) so re-indexing is idempotent.

</Pitfall>

<Quiz question="Why does Agent B use semantic search (vector embeddings) instead of keyword search to find relevant runbooks?">

<div>

Keyword search requires the query to use the same terms as the document. A log event saying "repeated authentication failures" wouldn't match a runbook that uses "brute force" — even though they describe the same thing.

Semantic search embeds both the query and the documents into a shared vector space where *meaning* is close, not just words. "Authentication failures from external IP" and "SSH brute force attack" end up near each other in that space, so the runbook is retrieved correctly.

This is critical for security documentation, which uses inconsistent terminology across vendors and teams.

</div>
</Quiz>
