## Answer Your Two Questions First

Before you open Gemini, write down — even just in your head — your answers to yesterday's two questions:

1. **Who is this for?**
2. **What's the ONE thing it has to do well?**

If you're building today's suggested project, here's a starting point:

<CodeLab title="Build Your v1">

```
Build a 5-question personality quiz that matches the person
taking it to one of four aircraft types — [pick four, e.g. a
nimble fighter jet, a long-haul airliner, a bush plane, a
helicopter] — based on how they'd want to travel, for a
first-time flyer who's curious but knows nothing about
aviation, styled like [your vibe]. Make the matched aircraft
reveal with a short, fun description when I click "See My
Match."
```

That's your PM lens already baked into the prompt: a specific audience ("a first-time flyer who's curious but knows nothing about aviation") and a specific single job ("match them to one aircraft type and reveal it clearly").

If you chose to iterate yesterday's boarding-pass project instead, apply the same lens to it: name who it's for, name the one thing it should do best, and prompt Gemini for one change that serves that job.

</CodeLab>

Run the loop — Describe, Build, Test, Refine, Repeat — just like yesterday. This time, as you test, check specifically for the three UX basics:

- Is there **one clear main action** on the page? (Clarity)
- Does something **visibly happen** when you click it? (Feedback)
- Does the **most important thing look the most important**? (Hierarchy)

<Pitfall title="Polishing Before It Works">

It's tempting to ask for a fancy animation before you've confirmed the core feature actually works end to end. Get the ONE thing — the quiz matching someone correctly, the boarding pass showing the right info — solid first. Polish (colors, animation, extra flair) is a refine step for *after* the core job works, not before.

</Pitfall>

## Submit & PM Pitch

Save your file using the same convention as yesterday: `name_day2_v1.html` if this is a new project, or the next version number if you iterated. Save it into the **Day 2** subfolder of the shared Drive (or commit it, if you're using GitHub this week).

Then, in **one sentence**, tell the room:

> "This is for [who], and it does [the one thing] well."

That's the whole pitch. No slides, no long explanation — just the PM lens, spoken out loud.

<Quiz question="Two students both built aircraft-matching quizzes today. One says: 'It's a quiz about planes.' The other says: 'It's for someone who's never flown before, and it matches them to an aircraft type in under a minute.' Which pitch shows stronger PM thinking, and why?">
<div>

**The second one — it names a specific audience and a specific, single job the build does well.**

"It's a quiz about planes" describes a topic, not a purpose. The second pitch answers both of today's core questions in one sentence: who it's for, and what it does well. That's the difference between describing what something *is* and describing what it's *for* — and it's the same gap between a feature and a value.

</div>
</Quiz>
